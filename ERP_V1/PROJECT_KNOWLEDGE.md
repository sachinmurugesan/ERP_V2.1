## 1. BUSINESS OVERVIEW

### What is HarvestERP?
A complete import management system for a business that **imports combine harvester spare parts from Chinese factories and sells them to Indian clients (farmers/dealers)**.

### Business Flow (Simple)
```
Indian Client places inquiry
    -> Company finds factory in China
    -> Factory quotes prices (via Excel)
    -> Company adds markup and creates Proforma Invoice (PI)
    -> Client approves PI and pays advance
    -> Company orders from factory
    -> Factory produces goods (~30 days)
    -> Goods shipped by sea (container)
    -> Arrives at Indian port
    -> Customs clearance
    -> Delivered to client via truck
    -> Order complete, final reconciliation
```

### Key Business Facts
- Communication with Chinese factories is via **Excel files** (not API, not email body - actual Excel attachments)
- Factory sends product images **embedded inside Excel cells** (not as separate attachments)
- Clients are in India, factories are in China
- Shipping is always by **sea container** (20FT, 40FT, 40HC)
- All customs clearance happens through **CFS (Container Freight Station)** - NOT direct port delivery
- Payment to factory is in **CNY/USD**, client pays in **INR** (multi-currency throughout)
- The sales team has **FULL authority** on payment decisions - no manager approval needed
- Production typically takes **30 days**
- Factory may NOT produce all ordered items - creating "unloaded items" that carry forward

## 2. USER PREFERENCES & DEVELOPMENT RULES

### MUST DO (Non-Negotiable)
| Rule | Details |
|------|---------|
| **Professional product** | This is NOT a prototype. Build production-ready code |
| **All manual triggers** | No automatic stage progression. Sales rep always clicks to move forward |
| **Full sales rep authority** | Sales rep can accept ANY payment amount (even zero), no restrictions, no manager approval |
| **All B/L data manual entry** | Bill of Lading details entered manually by user, no auto-generation |
| **Container number at sailing** | Container number entered manually only when ship has sailed |
| **Excel-based factory communication** | All factory data exchange through Excel files |
| **CFS only** | Always use CFS for container handling, no direct port delivery |
| **Product code can duplicate** | Same part code for different materials - differentiated by product name |
| **Product name is unique** | This is the real unique human identifier |
| **UUID is system identifier** | Internal unique ID for all records |
| **No pricing in product master** | Pricing is per-order, not stored in product catalog |
| **Separate service provider lists** | Freight Forwarder, CHA, CFS, Transport - all separate master data |
| **Exchange rates manual per transaction** | Default from settings, but user can override for each transaction |
| **Settings store default values** | Default exchange rates, markups, transit times - fetched when forms load |
| **Multiple containers per order** | One order can need multiple containers |
| **NO multiple orders per container** | Each container belongs to one order only |
| **Documents shared by factory** | Commercial Invoice and Packing List are shared by the factory (shipper), not generated by system |
| **Pallet numbering = simple** | Just pallet number (1-50) or "Loose" |
| **Factory payment before sailing** | Full payment to factory (loaded items only) triggers the sailing stage |
| **4 shipping docs from factory** | B/L, COO, CI, PL must be collected from factory before customs |
| **Track first, automate later** | New features start as tracking only (checkboxes/status). Functions added later |
| **SAILING = single combined stage** | LOADED + SAILED + ARRIVED are sub-phases of one SAILING stage, not separate stages |
| **CUSTOMS & CLEARANCE = single stage** | CUSTOMS + CLEARED combined into one stage with milestone logging |
| **Order number auto-gen at approval** | Order number auto-generated when DRAFT approved (→ PENDING_PI). Allow manual override |
| **GSTIN is optional** | Client must be registered, but GSTIN is not mandatory for all clients |
| **Markup at Stage 2 not Stage 1** | Markup applied when factory datasheet loaded (PENDING_PI), NOT at DRAFT stage |
| **Warehouse stock from container load** | Items added to warehouse stock list when loaded into container |
| **PI exported as Excel** | PI (Proforma Invoice) exported as Excel file. Ask format (Excel/PDF) on export |
| **Manual communication** | Factory and client communication is manual (WhatsApp/email). System generates documents, user shares them manually |
| **Documents block progression** | Required documents must be uploaded before "Next Stage" button is enabled (customs: BE, DO, OOC; transport: E-way Bill, Invoice) |
| **Customs = milestone logging** | Customs clearance tracked as status log/milestones. No complex functions. Just update status + upload documents |
| **Product-by-product delivery verification** | Every product must be individually checked at delivery: received? dimensions? quality? correct item? Each item gets ACCEPTED or OBJECTION status |
| **After-sales independent of order** | After-sales items have their own lifecycle (OPEN → IN_PROGRESS → RESOLVED → CLOSED). They continue even after the order is COMPLETED |
| **All products verified before COMPLETED** | Order cannot move to COMPLETED until every product verification is ACCEPTED or moved to After-Sales. No PENDING items allowed |
| **Per-item cost by factory price proportion** | Shared costs (freight, customs, transport) allocated to items proportional to their factory price value in INR |
| **Re-open completed orders** | COMPLETED orders can be re-opened for corrections. Track reason, timestamp, who. Must be re-closed. Status = COMPLETED_EDITING while editing |
| **Two export versions** | Internal (all data: factory price, markup, profit, costs) and Client-Facing (selling prices, payments, delivery only). Same DB, different templates |
| **Receivables tracking** | Outstanding payments after order close appear in global "Receivables" list. Track money owed per client/order |
| **After-sales impacts profit** | Credit/refund amounts from after-sales are subtracted from gross profit to show net profit. Both shown on completion screen |
| **Manufacturer code = primary product_code** | The manufacturer/factory part number (e.g., `W2.5E-01B-02-38X`) is the PRIMARY product_code. NOT the client barcode |
| **Client barcodes are per-client** | Each client has their own barcode for products. Store in `client_product_barcodes` mapping table (client_id, product_id, barcode_code). Match during Excel import |
| **Factory images saved to product gallery** | When parsing factory response Excel, extract all images and save per product. Products can have multiple images. Gallery view |
| **New products = PENDING_APPROVAL** | Unknown part codes from client Excel auto-create products with status `PENDING_APPROVAL`. Admin reviews from approval list. Don't block the order |
| **Duplicate codes = flag warning** | Same part code appearing multiple times in client Excel gets flagged. Admin decides: merge quantities or keep separate lines |
| **Client PI = clean, no factory data** | PI sent to client shows ONLY: Part Code, Description, Qty, Selling Price (INR), Total. NO factory prices, NO Chinese names, NO factory images, NO markup |

### MUST NOT DO
| Rule | Details |
|------|---------|
| **No automatic payment validation** | Do NOT restrict payment amounts or require minimum percentages |
| **No cheque clearance wait** | Accept cheques immediately, no "pending clearance" status |
| **No automatic stage transitions** | NEVER auto-advance order status. Always manual button click |
| **No payment variance thresholds** | Do NOT create approval workflows for payment variances |
| **No MANUAL product images in product form** | Product form does NOT have image upload. BUT factory-provided images from Excel ARE saved to product gallery automatically during factory response parsing |
| **No multiple orders in one container** | Keep it simple - one container = one order |
| **No auto-generated container numbers** | Container number comes from shipping line after vessel sails |
| **No live forex API (initially)** | Manual exchange rate entry with preset defaults |
| **No version history for PI** | Update existing PI, don't create versions |
| **No order number at DRAFT stage** | Order number auto-generated when DRAFT approved (→ PENDING_PI), can be manually overridden |
| **No price storage in product master** | Pricing is order-level, not product-level |

### PREFERRED PATTERNS
| Preference | Details |
|------------|---------|
| **Auto-detect everything** | Auto-detect paste format, Excel column layout, image positions |
| **Save often** | Auto-save drafts every 30 seconds |
| **Show warnings, don't block** | If quantity < MOQ, warn but allow. If payment is low, warn but allow |
| **Client portal for transparency** | Client can see order status, unloaded items, payment history via OTP portal |
| **Category-based markup** | Markup % set per category in Settings, can override per product in PI |
| **Keep Excel files 30 days** | Factory Excel uploads deleted after 30 days to save space |
| **Local storage first** | Store files locally, cloud migration planned for future |

### DEVELOPMENT ARCHITECTURE RULES
| Rule | Details |
|------|---------|
| **Use Python Enum for statuses** | All status fields (order status, payment status, etc.) use Python Enum classes. Prevents typos like "DRAFT" vs "draft". Provides autocomplete in IDE |
| **Store relative file paths only** | DB stores relative paths (e.g., `orders/ORD-2024-0001/documents/pi.xlsx`). Full path constructed from config. Enables Docker/cloud deployment |
| **Index critical query columns** | Add DB indexes on: `orders.status`, `orders.client_id`, `order_items.order_id`, `aftersales_items.status`, `payments.order_id` |
| **Exchange rate snapshot per order** | `exchange_rate` + `exchange_rate_date` stored on Order header. Historical accuracy even if rates change later |
| **Frontend state management** | Simple global state object for `currentOrder` data so tab navigation doesn't refetch API every time |
| **Audit trail for re-opens** | Every order re-open logged with: who, when, reason. `reopen_count` tracked on Order model |

### Key Learnings
1. **Always ask before assuming** - Don't assume business rules. The user knows their business.
2. **Flexibility over restriction** - When in doubt, give the user MORE control, not less.
3. **Manual over automatic** - In import/export business, humans make decisions. System records and tracks.
4. **Simple over complex** - Pallet numbering is just 1-50 or "Loose". Don't overthink it.
5. **Real-world first** - Factory communication is via Excel and WhatsApp, not APIs. Design for real workflows.
6. **Track first, automate later** - User prefers simple tracking (checkboxes, statuses) first. Complex functions added later when needed.
7. **Combine related stages** - If phases are part of one continuous process, keep them as ONE stage with internal sub-phases. Don't split into separate stages unnecessarily.
8. **Payment triggers movement** - Factory payment triggers sailing. This is real-world: factory releases goods after payment.
9. **Documents gate progression** - Required documents (BE, DO, OOC, E-way Bill, Invoice) must be uploaded before "Next" button works. Documents are the proof of milestone completion.
10. **Charges are for internal calculation** - Fetch all costs (CHA, CFS, duty, transport) at the end for landed cost/profit calculation. Don't build complex charge-tracking functions.
11. **Stock starts at loading** - Warehouse stock list populated when items are loaded into container. This is when goods become "our inventory."
12. **Excel is primary export format** - PI and other documents exported as Excel primarily. Always ask format (Excel/PDF) on export.
13. **Delivery = product-by-product verification** - Don't simplify delivery to just a timestamp. Every product must be individually checked for: receipt, dimensions, quality, correctness. This is the quality gate.
14. **After-sales is independent** - After-sales items have their own lifecycle separate from the order. An order can be COMPLETED while after-sales issues are still OPEN. They're linked but not dependent.
15. **Nothing is truly permanent** - Even "completed" orders may need corrections. Always provide a re-open mechanism with audit trail (who, when, why).
16. **Internal vs external views** - Same data, different audiences. Factory prices/markup/profit are NEVER shown to clients. Always design two export templates from the same database.
17. **Cost allocation by value** - When splitting shared costs across items, use factory price proportion. It's the most fair and standard business practice for import costing.
18. **Track what's owed** - Outstanding payments after order close need a global "Receivables" view. Don't lose track of money just because the order workflow is done.
19. **After-sales affects profit** - Show the impact of after-sales credits/refunds on profit calculation. Net profit = gross profit minus after-sales costs. Track separately but display together.
20. **Handle ambiguity explicitly** - When the same code maps to multiple records (duplicate part codes), don't silently pick one or crash. Show all options and let the user choose.
21. **Validate at gates, not just at save** - Stage transitions are the critical validation points. Every "Next Stage" button must check that ALL required data from current stage is complete.
22. **Snapshot mutable data** - Exchange rates, markups, prices change over time. Store the rate/value at time of transaction on the order itself, not just a reference to current settings.
23. **Re-read business rules before proposing solutions** - Always cross-check proposed solutions against MUST DO rules. Mistake #37 was the SAME as Mistake #5 because I didn't re-read the rules before answering queries.
24. **Stage-specific features in stage-specific views** - Don't add pricing, markup, or stage-specific workflows to the generic OrderDetail view. Each stage that needs special functionality gets its own dedicated view component.
25. **Label clarity matters** - If a user asks "what does this field mean?", the label is wrong. Use domain-specific labels ("Factory Currency" not "Currency") and add helper text for non-obvious fields.
26. **Copy from previous order > default values** - For repeat orders, a "Copy from previous order" pattern is better than storing defaults. It respects per-order pricing rules while reducing data entry.
27. **Real Excel files reveal real requirements** - The actual Client Request and Factory Respond Excel files showed the true workflow: client sends 3 columns (barcode + manufacturer code + qty), factory returns 9 columns + images. Always analyze real data before building parsers.
28. **Client barcodes are per-client identifiers** - Column A in client Excel is NOT a universal product code. It's a client-specific barcode. The same product has different barcodes for different clients. Need a mapping table.
29. **Factory images ARE stored per product** - Despite earlier "no images in catalog" rule, factory Excel responses contain product images (71 images for 56 items). These get extracted and saved to a product image gallery. The rule should be: "No manual upload in product form, but factory images ARE saved."
30. **New products auto-created as PENDING_APPROVAL** - When client sends a part code not in our database, create the product with status PENDING_APPROVAL. Admin reviews and approves from a list. Don't block the order.
31. **Duplicate part codes get flagged, not auto-merged** - When same part code appears twice in client Excel (e.g., row 7 and row 46 both `10700BDAS`), show warning to admin. Let them decide: merge quantities or keep as separate lines.
32. **Clean PI for client** - Client PI shows ONLY: Part Code, Description, Qty, Selling Price (INR), Total. NO factory prices, NO Chinese names, NO factory images, NO markup percentages. Two audiences = two views.

### Architecture Decisions
- **SPA (Single Page Application)**: Vue.js handles all routing via Vue Router
- **REST API**: Backend exposes `/api/` endpoints, frontend calls via Axios
- **Component-Based UI**: Each stage, form, table = reusable Vue component
- **Pinia Store**: Global state for currentOrder, user session, settings cache
- **No Authentication (Phase 1)**: Basic auth to be added later
- **File Storage**: Local filesystem initially, cloud (S3/Azure) in future
- **Soft Deletes**: Use `deleted_at` timestamp instead of hard deletes
- **Enum for Statuses**: Python Enum classes for all status fields
- **Relative File Paths**: DB stores relative paths, full path from config

## 6. WORKFLOW - COMPLETE SPECIFICATION

### Stage Count Note
Originally 17 internal statuses, now grouped into logical combined stages for clarity:
- Stage 4 combines ADVANCE_PENDING + ADVANCE_RECEIVED
- Stage 5-9 combines FACTORY_ORDERED + PRODUCTION_60/80/90/100
- Stage 11 combines LOADED + SAILED + ARRIVED into single "SAILING" stage
- Stage 12 combines CUSTOMS + CLEARED into single "CUSTOMS & CLEARANCE" stage

### Workflow Overview
```
STAGE 1:   DRAFT                  - Sales rep creates order with product list
STAGE 2:   PENDING_PI             - Send inquiry to factory, receive quotation via Excel
STAGE 3:   PI_SENT                - Create PI with markup, send to client for approval
STAGE 4:   ADVANCE_PENDING/       - Client pays advance, sales rep accepts any amount
           ADVANCE_RECEIVED         (combined single stage)
STAGE 5-9: PRODUCTION             - Factory produces goods over 30 days
           (FACTORY_ORDERED,        Packing list review, loaded vs unloaded items
            PRODUCTION_60/80/90/100)
STAGE 10:  BOOKED                 - Book container, assign service providers
STAGE 11:  SAILING                - Full factory payment, container loaded, vessel sailing,
           (LOADED→SAILED→ARRIVED)  collect documents from factory, track voyage to arrival
STAGE 12:  CUSTOMS & CLEARANCE    - Customs filing, clearance milestones, document uploads,
           (CUSTOMS→CLEARED)        transport arrangement with E-way Bill & Invoice
STAGE 13:  DELIVERED              - Goods transported and delivered to client
STAGE 14:  COMPLETED              - Final reconciliation, order closed
```

### STAGE 2: PENDING_PI (Factory Quotation & Selling Price)
**Purpose**: Send product list to Chinese factory, receive price quotation via Excel. Apply markup to generate selling prices for client.

**FRONTEND**:
- **Export product list** as Excel → user manually sends to factory via WeChat/email
- **Upload factory Excel quotation** (max 200MB) back into system
- Auto-extract product images embedded in Excel cells
- Auto-detect which column contains images, match by row position
- **Factory Price Table** displayed after upload:
  - Columns: Product Code, Name, Qty, Factory Price (CNY), Images
  - All prices loaded from factory Excel
- **"Convert to Selling Price" Button** on the table:
  - When clicked: Adds new columns to the SAME table:
    - Markup % (default from category Settings, editable per product)
    - Selling Price (CNY) = Factory Price × (1 + Markup%)
    - Selling Price (INR) = Selling Price CNY × Exchange Rate
  - User can adjust markup per product row
  - All selling prices calculated and shown inline on same sheet/table
- One factory per order

**BACKEND**:
- Parse uploaded Excel: Read factory prices per product
- Extract embedded images from Excel cells (openpyxl)
- Match images to products by row anchor position
- Save images: `/images/{order_id}/{product_code}_{version}.png`
- Store factory prices in OrderItem records
- When "Convert to Selling Price" triggered:
  - Load category markup defaults from Settings
  - Calculate selling price per item
  - Store markup% and selling_price in OrderItem

**Image Extraction Logic**:
```
1. Read Excel with openpyxl
2. Find images in worksheet._images
3. Match each image to row by anchor position
4. Save image: /images/{order_id}/{product_code}_{version}.png
5. Link to product in this order
```

**BUSINESS RULES**:
- Communication via Excel ONLY (not API) - manual send via WeChat/email
- Images embedded IN Excel cells (not separate files)
- Auto-detect image column
- Warning for missing images (don't block)
- One factory per order
- Keep image version history (factory may send updated Excel)
- Excel files deleted after 30 days, images kept
- **Markup applied HERE** (not at DRAFT, not at PI_SENT):
  - Default markup% from category Settings
  - Editable per product in the selling price table
  - Selling price = factory price × (1 + markup/100)
  - Both CNY and INR selling prices shown

**NEXT STAGE TRIGGER**: Sales rep confirms selling prices are set → Clicks "Generate PI" → Status = PI_SENT
- **VALIDATION**: Every order item MUST have `selling_price_inr > 0` before "Generate PI" is enabled. If any item has factory price but no selling price (user forgot to click "Convert to Selling Price"), BLOCK with warning: "X items have no selling price. Click 'Convert to Selling Price' first."

### STAGE 11: SAILING (Combined: LOADED + SAILED + ARRIVED)
**Purpose**: After full factory payment is made, track the entire shipping journey - from container loading at factory, through ocean voyage, to arrival at Indian port. Collect all shipping documents from factory (shipper) before arrival for customs preparation.

**IMPORTANT**: This is a **tracking-only stage**. No complex functions or automations. Just record data and track document collection progress. Functionality can be added later.

#### **11.1 Stage Entry: Factory Final Payment**
Before container starts sailing, full payment to factory must be made for the **confirmed packing list items only** (loaded items, NOT unloaded items).

**Factory Payment**:
```
Payment to Factory:
- Amount: Based on confirmed packing list (loaded items only)
- Currency: CNY or USD (as per factory agreement)
- Exchange rate: Manual entry (default from Settings)
- Payment method: Bank Transfer (TT - Telegraphic Transfer)
- Payment reference: Bank transaction reference
- Payment date: Date of transfer
```

**Rule**: Factory payment = Value of LOADED items only. Unloaded items are NOT paid for (they were already excluded in packing list confirmation at Stage 5-9).

**Trigger to Start Sailing**: Sales rep clicks "Confirm Factory Payment & Start Sailing" -> Status = SAILING

**Phase 3: ARRIVED** (Container at Indian Port)
- Actual Arrival Date: Date picker (manual)

- Notes: Text area

#### **11.3 Document Collection from Factory (Shipper)**
**CRITICAL**: Before the container arrives at Indian port, ALL shipping documents must be collected from the factory (shipper). These are required for customs filing.

**Required Documents from Factory**:

| # | Document | Abbreviation | Description | Status Tracking |
|---|----------|-------------|-------------|-----------------|

#### **13.5 BUSINESS RULES**

1. **Every product must be verified**: Cannot proceed to COMPLETED until all line items are either ACCEPTED or have an OBJECTION (moved to after-sales). No "Pending" items allowed.
2. **Objection requires evidence**: When marking objection, description and photos are MANDATORY. This is proof for factory claims.
3. **After-Sales is a separate ongoing process**: Moving item to after-sales does NOT block order completion. After-sales issues continue independently.
4. **Objection types cover all scenarios**:
   - `DIMENSION_MISMATCH` - Size/measurement doesn't match specification
   - `WRONG_PRODUCT` - Completely different product than ordered
   - `QUALITY_ISSUE` - Product quality below standard/expectation

   - `SHORTAGE` - Received less quantity than what was loaded
   - `OTHER` - Any other issue with free-text description
5. **Shortage at delivery vs unloaded items**: These are DIFFERENT:
   - Unloaded items (Stage 5-9) = Factory didn't produce, items never left China
   - Shortage at delivery (Stage 13) = Items were loaded but some missing/lost during transit
6. **After-Sales list is visible globally**: Dedicated "After-Sales" top-level menu showing all open items across all orders
7. **Final payment is optional for progression**: Sales rep can proceed to COMPLETED even with outstanding balance
8. **Delivery photos recommended**: For accepted items too, as proof of successful delivery

#### **14.4 DATA**

**Landed cost = calculated on-the-fly** by fetching from:
- `factory_payments` → Factory cost (Stage 11)
- `shipments` → Freight charges (Stage 10)
- `clearance_charges` → Customs/CHA/CFS costs (Stage 12)
- `payments` → Client payments (Stage 4 + Stage 13)
- `aftersales_items` → Credit/refund amounts for after-sales impact

**Per-Item Cost Allocation Formula**:
```
Shared Costs = Freight + Insurance + CHA + CFS + THC + Transport + Other Charges
(Customs duties BCD+SWS+IGST are also shared costs)

For each item:
  item_factory_value_inr = factory_price_per_unit × quantity × exchange_rate
  total_factory_value_inr = SUM of all items' factory_value_inr
  item_proportion = item_factory_value_inr / total_factory_value_inr

  item_shared_cost = Shared Costs × item_proportion
  item_landed_cost = item_factory_value_inr + item_shared_cost
  item_landed_cost_per_unit = item_landed_cost / quantity

  item_profit = (selling_price_per_unit × quantity) - item_landed_cost
  item_margin = item_profit / (selling_price_per_unit × quantity) × 100
```

**After-Sales Profit Impact**:
```
For each item with after-sales resolution:
  item_net_profit = item_profit - aftersales_credit_amount

Order level:
  gross_profit = total_client_payment - total_landed_cost
  total_aftersales_credits = SUM of all resolved after-sales credit/refund amounts
  net_profit = gross_profit - total_aftersales_credits
```

#### **14.5 BUSINESS RULES**

1. **All Products Must Be Verified**: Cannot close order if any product is still "Pending" verification from Stage 13
2. **After-Sales Continues Independently**: Closing order does NOT close after-sales items. They have their own lifecycle (OPEN → IN_PROGRESS → RESOLVED → CLOSED)
3. **After-Sales Impacts Profit Display**: After-sales credit/refund amounts are subtracted from gross profit to show net profit. Tracked separately but impact shown on same screen.
4. **Sales Rep Decides Closure**: Can close with outstanding payment balance. Sales rep has full authority.
5. **Landed Cost = Auto-Fetched**: All costs fetched from data already entered in previous stages. No re-entry.
6. **Per-Item Cost Allocation = By Factory Price**: Shared costs (freight, customs, transport, etc.) distributed across items proportional to their factory price value (INR). Higher-value items get proportionally more cost allocated.
7. **IGST Credit Independent**: Can update IGST claim status even after order is closed
8. **Re-open Allowed**: COMPLETED orders CAN be re-opened for corrections (charge edits, payment updates, notes). Status temporarily becomes "COMPLETED_EDITING". Must be re-closed. Tracks reason + timestamp.
9. **Receivables Tracking**: Orders closed with outstanding balance appear in global "Receivables" list. Filtered by client, order, amount. Helps track money still owed.
10. **Warehouse Stock Updated**: All items for this order marked as DELIVERED in warehouse stock
11. **After-Sales Visible**: After-sales items shown in completion summary and linked to global After-Sales menu
12. **Two Export Versions**:
    - **Internal**: Full data — factory prices, markup%, landed cost breakdown, per-item profit, after-sales details, all charges
    - **Client-Facing**: Only selling prices, payment summary, delivery confirmation, after-sales status. NO factory prices, NO markup, NO internal costs, NO profit data
13. **Same Database, Two Views**: No separate storage. Export endpoint applies different templates to same data based on `version` parameter.

## 9. SERVICE PROVIDERS

### 4 Types of Service Providers

| # | Provider | Role | When Selected | When Used |
|---|----------|------|---------------|-----------|
| 1 | **Freight Forwarder** | Books vessel, handles ocean freight | Stage 10 (BOOKED) | Stage 10-11 (booking to sailing) |
| 2 | **CHA** | Files customs documents, handles clearance | Stage 10 (BOOKED) | Stage 12 (CUSTOMS & CLEARANCE) |
| 3 | **CFS** | Stores and de-stuffs container at port | Stage 10 (BOOKED) | Stage 11-12 (arrival to cleared) |
| 4 | **Transport** | Delivers goods from CFS to client | Stage 10 or 12 | Stage 12-13 (clearance to delivery) |

### Rules
- Each type has its own master data table
- Freight Forwarder + CHA + CFS = MANDATORY at booking
- Transport = OPTIONAL at booking (can select at Stage 12 during clearance)
- CHA must match Port of Discharge
- All have: company_name, contact_person, email, phone, address, gstin, is_active

## 10. MODULE BUILD ORDER

### Why Module-Level (Not Stage-Level)
Here's the module-level build order based on dependencies — what needs to exist before other things can work. Building by module is smarter than building stage-by-stage because stages depend on modules. A stage like "BOOKED" needs Service Providers, Shipments, and Payments all working — those are separate modules. Build the foundations first, then stages naturally work.

### Dependency Tree
```
Level 0 (Foundation) ──→ Level 1 (Master Data) ──→ Level 2 (Order Engine)
                                                        │
                                          ┌─────────────┼─────────────┐
                                          ▼             ▼             ▼
                                    Level 3         Level 4       Level 5
                                  (Pricing/PI)    (Payments)   (Production)
                                          │             │             │
                                          └──────┬──────┘             │
                                                 ▼                    │
                                           Level 6 ◄──────────────────┘
                                        (Shipping/Booking)
                                                 │
                                                 ▼
                                           Level 7
                                        (Customs/Clearance)
                                                 │
                                                 ▼
                                           Level 8
                                       (Delivery/Verification)
                                                 │
                                                 ▼
                                           Level 9
                                      (After-Sales/Completion)
                                                 │
                                                 ▼
                                          Level 10
                                      (Dashboard/Reports)

                                          Level 11
                                       (Client Portal)
                                    [independent after Level 3]
```
