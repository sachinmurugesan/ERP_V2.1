# HarvestERP — Security Verification of Top 3 Leaks (D-004)

**Date:** 2026-04-21
**Source audit:** [SECURITY_AUDIT.md](./SECURITY_AUDIT.md)
**Method:** Static code trace — followed each claimed leak from the router registration in `main.py` through the dependency chain into the endpoint handler. No runtime testing performed. Where a runtime test would strengthen confidence beyond static trace, it is called out.
**Purpose:** Filter false positives before authoring the Day-2 hotfix. Leaks that pass this verification are real attack surfaces; leaks that fail it are corrected here.

---

## Leak 1 — Dashboard `/recent-orders/` and `/active-shipments/` leak cross-tenant CNY totals

**Audit claim:** Any authenticated user — including CLIENT and FACTORY tokens — can read `total_value_cny` aggregated across all tenants via these two endpoints.

### Code trace

1. [`main.py:190-191`](../../backend/main.py#L190) mounts dashboard router:
   ```python
   app.include_router(dashboard.router, prefix="/api/dashboard", tags=["Dashboard"],
                      dependencies=[Depends(get_current_user)])
   ```
   Only JWT validation — no role check, no user-type gate.
2. [`dashboard.py:67-93`](../../backend/routers/dashboard.py#L67) `get_recent_orders(db: Session = Depends(get_db))` — **no `current_user` parameter**. The function fetches `db.query(Order).filter(Order.deleted_at.is_(None))` with no `client_id` or `factory_id` scoping.
3. [`dashboard.py:13-21`](../../backend/routers/dashboard.py#L13) `_compute_total_cny(order)` returns `sum(i.factory_price * i.quantity for i in order.items if i.status == "ACTIVE")`. The returned field `total_value_cny` is the real factory-paid cost in Chinese Yuan.
4. No role-based stripping via `filter_for_role` is applied — the handler returns a plain dict.

### Test scenario

**Prereq:** A valid CLIENT user token (generated by logging into any client account via `POST /api/auth/login`). Call this `$CLIENT_TOKEN`.

**Request:**
```bash
curl -sS -H "Authorization: Bearer $CLIENT_TOKEN" \
     https://<host>/api/dashboard/recent-orders/
```

**Expected response (the leak):**
```json
[
  {
    "id": "ord-abc-123",
    "order_number": "SSP/25-26/03/0001",
    "po_reference": "…",
    "client_name": "Some Other Client Pvt Ltd",
    "factory_name": "Chongqing Harvester Parts Co.",
    "status": "PRODUCTION_80",
    "stage_number": 7,
    "stage_name": "…",
    "total_value_cny": 482350.00,
    "created_at": "…"
  },
  …  // up to 10 orders, cross-tenant
]
```

The caller — who is a CLIENT user of Tenant A — reads `total_value_cny` for orders belonging to Tenant B, Tenant C, etc. Per the DECISIONS.md D-004 matrix, `view_supplier_chinese_yuan_pricing` is SUPER_ADMIN-only. Exposure here is to **every authenticated role including external CLIENT and FACTORY users**.

**The same attack works against `/api/dashboard/active-shipments/`** ([`dashboard.py:96-129`](../../backend/routers/dashboard.py#L96)) — identical pattern, identical `_compute_total_cny` helper, same absence of `current_user` / scoping.

### Confidence

**CONFIRMED LEAK.**

The code path is deterministic and unambiguous: valid JWT → router accepts → handler has no scoping → full cross-tenant response. No conditional gates exist between the JWT check and the response. A runtime test would produce the documented behaviour with 100% probability given a working token of any role.

**One minor uncertainty worth runtime-verifying:** whether the JWT of a CLIENT user is actually accepted by `get_current_user` without any additional type gate. Per [`security.py:205-232`](../../backend/core/security.py#L205), the function decodes the token and returns a `CurrentUser` regardless of `user_type` — so yes, CLIENT tokens pass. A 30-second curl with a real client token would remove the last bit of residual doubt, but the code tells the same story.

## Leak 2 — Factory ledger family (5 endpoints) leaks factory CNY obligations to ADMIN

**Audit claim:** An ADMIN token can read the factory ledger, factory credits, factory payments, apply-factory-credit, and payment audit logs — despite the DECISIONS.md D-004 matrix excluding ADMIN from `view_factory_ledger` (which is SUPER_ADMIN + FINANCE only).

### Code trace

1. [`main.py:218-219`](../../backend/main.py#L218) mounts the main finance router:
   ```python
   app.include_router(finance.router, prefix="/api/finance", tags=["Finance"],
                      dependencies=[Depends(require_finance)])
   ```
2. [`security.py:255`](../../backend/core/security.py#L255):
   ```python
   require_finance = require_role([UserRole.ADMIN, UserRole.FINANCE])
   ```
   The allowed-roles list explicitly includes ADMIN.
3. [`security.py:238-249`](../../backend/core/security.py#L238) — `require_role`:
   ```python
   async def _check_role(current_user: CurrentUser = Depends(get_current_user)):
       if UserRole.ADMIN in current_user.roles:
           return None                      # (a) hard-coded ADMIN bypass
       if not current_user.has_any_role(allowed_roles):
           raise HTTPException(status_code=403, …)
       return None                          # (b) ADMIN also in allow-list
   ```
   ADMIN passes via two independent paths. Either path would be sufficient on its own.
4. The five factory-facing endpoints return CNY/USD factory-side data:
   - [`finance.py:1475`](../../backend/routers/finance.py#L1475) `/factory-ledger/{factory_id}/` — per-order CNY totals computed from `factory_price × quantity` (line 1529) and USD conversion.
   - [`finance.py:1834`](../../backend/routers/finance.py#L1834) `/factory-ledger/{factory_id}/download/` — downloadable version.
   - [`finance.py:1384`](../../backend/routers/finance.py#L1384) `/factories/{factory_id}/credits/` — factory credit amounts.
   - [`finance.py:724`](../../backend/routers/finance.py#L724) `/orders/{order_id}/factory-payments/` — factory payment history.
   - [`finance.py:952`](../../backend/routers/finance.py#L952) `/orders/{order_id}/payment-audit-log/` — before/after JSON snapshots of factory payments.

### Test scenario

**Prereq:** A valid ADMIN user token. Call this `$ADMIN_TOKEN`. Any factory ID from the DB, call it `$FACTORY_ID`.

**Request:**
```bash
curl -sS -H "Authorization: Bearer $ADMIN_TOKEN" \
     https://<host>/api/finance/factory-ledger/$FACTORY_ID/
```

**Expected response (the leak):**
```json
{
  "entries": [
    {
      "date": "2026-02-14",
      "order_number": "SSP/25-26/02/0017",
      "remark": "Factory order for SSP/25-26/02/0017",
      "debit": 1842500.00,
      "credit": 0,
      "amount_foreign": 218750.00,      // ← CNY — real factory cost
      "currency": "CNY",
      "exchange_rate": 8.42,
      "amount_usd": 21932.14,
      …
    },
    … more entries …
  ],
  "summary": {
    "total_debit": 5872350.00,
    "total_credit": 4210000.00,
    "net_balance": 1662350.00,
    "total_debit_usd": …,
    …
  },
  "factory_name": "Chongqing Harvester Parts Co.",
  "factory_id": "fct-…"
}
```

The ADMIN sees the real CNY cost book per factory — which the D-004 matrix marks SUPER_ADMIN + FINANCE only. The same ADMIN token extracts equivalent CNY/USD data from the other four endpoints in the family.

### Confidence

**CONFIRMED LEAK.**

Three independent points of verification all agree:
1. The role list in `require_finance` explicitly names ADMIN.
2. `require_role` has an unconditional ADMIN bypass ahead of the allow-list check — this would let ADMIN through even if step 1 were wrong.
3. The handlers have no secondary inline role check downstream of the router dependency.

There is no conditional path by which an ADMIN request would be denied. No runtime test needed to establish the leak exists.

**Matrix nuance for disclosure:** ADMIN reading the factory ledger is a leak *only* if D-004's matrix is accepted as-written (`view_factory_ledger = [SUPER_ADMIN, FINANCE]`). If the owner decides during sign-off that ADMIN should have this capability, this entire leak dissolves into OK. Flag this as the first matrix-confirmation question for Sachin.

## Leak 3 — PI downloads serve raw Excel with factory pricing

**Audit claim (as written in SECURITY_AUDIT.md):** "FINANCE/OPERATIONS can download any PI containing `factory_price` on REGULAR clients; the Excel file is not stripped per-role" and "OPERATIONS can download PIs containing `factory_price`; no per-order scoping — any order's PI is readable".

**Verification question as posed:** *"can a request with no auth at all retrieve a raw Excel?"*

### Part A — The "no auth at all" framing is REFUTED

The four PI-download endpoints are behind router-level auth dependencies:

| Endpoint | Router mount in `main.py` | Effective requirement |
|---|---|---|
| `GET /api/orders/{order_id}/download-pi/` ([`orders.py:4316`](../../backend/routers/orders.py#L4316)) | `dependencies=[Depends(get_current_user)]` ([main.py:194-195](../../backend/main.py#L194)) | valid JWT required |
| `GET /api/orders/{order_id}/download-pi-with-images/` ([`orders.py:4336`](../../backend/routers/orders.py#L4336)) | same | valid JWT required |
| `GET /api/excel/download-pi/{order_id}/` ([`excel.py:602`](../../backend/routers/excel.py#L602)) | `dependencies=[Depends(require_operations)]` ([main.py:206-207](../../backend/main.py#L206)) | ADMIN / OPERATIONS / SUPER_ADMIN |
| `GET /api/excel/download-pi-with-images/{order_id}/` ([`excel.py:618`](../../backend/routers/excel.py#L618)) | same | ADMIN / OPERATIONS / SUPER_ADMIN |

Router-level dependencies run *before* any endpoint handler. A request without an Authorization header is rejected with `401 Authorization header missing` by [`security.py:211-217`](../../backend/core/security.py#L211) inside `get_current_user`, which is transitively required by all four routes.

**Test to demonstrate the refutation:**
```bash
curl -sS -i https://<host>/api/excel/download-pi/some-order-id/
# Expected: HTTP/1.1 401 Unauthorized
# Body: {"detail":"Authorization header missing"}
```

**Confidence: REFUTED** — "no auth at all" does not retrieve a PI. Plain static trace is conclusive.

### Part B — The real leaks SECURITY_AUDIT.md documents are still CONFIRMED

The audit did not actually claim "no auth at all" — it claimed two separate leaks that the verification question conflated. Verifying each:

#### B1 — Cross-order access with no ownership check (`excel.py` variants)

[`excel.py:602-615`](../../backend/routers/excel.py#L602) `download_pi` signature:
```python
@router.get("/download-pi/{order_id}/")
def download_pi(order_id: str, db: Session = Depends(get_db)):
    pi = db.query(ProformaInvoice).filter(ProformaInvoice.order_id == order_id).first()
    …
    return FileResponse(path=pi.file_path, …)
```
No `current_user` parameter. The router-level `require_operations` guarantees the caller has ADMIN / OPERATIONS / SUPER_ADMIN, but **any such user can request any `order_id`**. No `get_scoped_query` call, no call to `_get_order_for_pi` (which lives in the orders router and does ownership checks — this endpoint bypasses it).

**Test scenario:**
```bash
# $OPS_TOKEN is an OPERATIONS user with no legitimate access to ord-abc-123
curl -sS -o pi.xlsx -H "Authorization: Bearer $OPS_TOKEN" \
     https://<host>/api/excel/download-pi/ord-abc-123/
# Expected: HTTP 200 + binary .xlsx file
```

**Leaked data inside `pi.xlsx`:** columns `factory_price`, `markup_percent`, `selling_price_inr` per line item — the Excel is generated with all internal columns and served as-is. No column stripping occurs at download time.

**Confidence: CONFIRMED LEAK.** The function signature lacks any ownership check. The router auth enforces role but not scope.

*Nuance:* in a tenant with a single-OPERATIONS-user model, this leak is equivalent to that operator's existing data access — so the practical attack window is narrow. But any migration to a multi-operator setup (or a compromised ops token) expands the blast radius immediately.

#### B2 — Pricing visibility via PI contents even with legitimate ownership

[`orders.py:4316`](../../backend/routers/orders.py#L4316) `client_download_pi` has `_get_order_for_pi` ownership scoping, so a CLIENT sees only their own order. **But an internal FINANCE or OPERATIONS token hitting this same endpoint gets any order's PI, with `factory_price` columns intact.** The Excel is streamed as a `FileResponse` — the handler has no hook to strip per-role columns.

**Test scenario:**
```bash
# $FINANCE_TOKEN is a FINANCE user
curl -sS -o pi.xlsx -H "Authorization: Bearer $FINANCE_TOKEN" \
     https://<host>/api/orders/ord-abc-123/download-pi/
# Expected: HTTP 200 + binary .xlsx containing factory_price column
```

Per the D-004 matrix, `view_factory_real_pricing` is SUPER_ADMIN-only and `view_factory_quoted_pricing` is SUPER_ADMIN + ADMIN. FINANCE should not see `factory_price` under either key.

**Confidence: CONFIRMED LEAK.** Static trace is definitive for the code path; the remaining question ("does the on-disk PI file actually contain a `factory_price` column?") is answerable with a single runtime open of the generated file — which the PI generator at [`excel.py:618-`](../../backend/routers/excel.py#L618) does produce, per manual code review of the sheet-building logic. If a runtime test is desired, opening any existing `pi.file_path` from the DB confirms the column set in under 10 seconds.

### Overall Leak 3 verdict

| Sub-claim | Verdict |
|---|---|
| "No auth at all retrieves a raw Excel" | **REFUTED** — 401 at router-level dependency. |
| `excel.py:602/618` — missing per-order ownership, OPERATIONS can download any PI | **CONFIRMED LEAK** |
| `orders.py:4316/4336` — FINANCE / OPERATIONS can download PIs containing factory pricing fields they should not see | **CONFIRMED LEAK** |
